1.3 Reach Programs

This document describes the structure and content of Reach programs, including their syntactic forms, the Reach standard library, and the standards of valid programs.

1.3.1 Validity

Reach imposes further restrictions on syntactically well-formed programs. These restrictions are described throughout this manual using the term valid to refer to constructions that obey the restrictions and invalid to refer to constructions that do not obey them.

It is always invalid to use a value with an operation for which it is undefined. For example, 1 + true is invalid. In other words, Reach enforces a static type discipline.

1.3.2 Source Files

Reach source files are either executables or libraries. They are traditionally given the file extension rsh, e.g. "dao.rsh".

Reach executables start with 'reach 0.1 exe'; and are followed by a sequence of imports, participant definitions, identifier definitions, and a main function. The evaluation of the execution is that of the main function.

Reach libraries start with 'reach 0.1 lib'; and are followed by a sequence of imports and identifier definitions.

1.3.3 Comments

// single-line comment
/* multi-line
 * comment
 */

Comments are text that is ignored by the compiler. Text starting with // up until the end of the line forms a single-line comment. Text enclosed with /* and */ forms a multi-line comment. It is invalid to nest a multi-line comment within a multi-line comment.

1.3.4 Imports

import "games-of-chance.rsh";

When a Reach source file, X, contains an import, written import "LIB.rsh";, then the path "LIB.rsh" must resolve to a file which is a library. The definitions located in "LIB.rsh" are included in the set of definitions associated with X.

The path given to an import may include .. to specify files outside the current directory.

1.3.5 Participant Definitions

const Alice = newParticipant();

Reach executables may contain participant definitions, which are written:

const PARTICIPANT = newParticipant();

Such a definition defines a participant named PARTICIPANT.

1.3.6 Types

A base type is either:

  • address, which denotes an account address; or,

  • uint256, which denotes an unsigned integer of 256 bits; or,

  • bool, which denotes a boolean; or,

  • bytes, which denotes a string of bytes.

A type is either a base type or a statically-sized homogeneous array of base type values, written BASE[EXPR] where BASE is a base type and EXPR is an expression that statically evaluates to a natural number. For example, the following are all types:

address
bool
bool[4]
uint256[4 + 5]
uint256[3 * 7 + 99]

1.3.7 Identifier Definitions

An identifier definition is either a value definition, enumeration, or function definition. Each of these bind identifiers.

—
const DELAY = 10;
const [ Good, Bad ] = [ 42, 43 ];

A value definition is written const LHS = RHS; where LHS is either a single identifier, e.g. isDelicious, or an array of identifiers, e.g. [ bestSushi, mediumestSushi, worstSushi ], and RHS is an expression. RHS must evaluate to as many values as there are identifiers in LHS. Those values are available as the corresponding identifiers in the rest of the program.

—
const [ isHand, ROCK, PAPER, SCISSORS ] = makeEnum();

An enumeration, written const [ ENUM, OPTION_0, ..., OPTION_n ] = makeEnum();, defines the identifiers OPTION_0 through OPTION_n as unique natural numbers and ENUM as a function which abstracts over one numeric argument and returns true if and only if it is one of these natural numbers.

—
function randomBool() {
  return (random() % 2) == 0; };

A function definition, written function FUN(ARG_0, ..., ARG_n) BLOCK;, defines FUN as a function which abstracts its function body, the block BLOCK, over the identifiers ARG_0 through ARG_n.

—
function main() {
  return 42; };

A main function is a function definition with the name main and no arguments. The evaluation of the main function is the same as its function body.

—

All identifiers in Reach programs must be unbound at the position of the program where they are bound, i.e., it is invalid to shadow identifiers with new definitions. For example,

const x = 3;
const x = 4;

is invalid. This restriction is independent of whether a binding is only known to a single participant. For example,

Alice.only(() => {
  const x = 3; });
Bob.only(() => {
  const x = 3; });

is invalid.

1.3.8 Blocks

{ return 42; }
{ const x = 31;
  return x + 11; }
{ if ( x < y ) {
    return "Why";
  } else {
    return "Ecks"; } }

A block is a sequence of statements surrounded by braces, i.e. { and }.

1.3.9 Statements

There are a large variety of different statements in Reach programs.

Each statement affects the meaning of the subsequent statements, which is called its tail. For example, if {X; Y; Z;} is a block, then X’s tail is {Y; Z;} and Y’s tail is {Z;}.

Distinct from tails are continuations which include everything after the statement. For example, in {{X; Y;}; Z;}, X’s tail is just Y, but its continuation is {Y;}; Z;.

Tails are statically apparent from the structure of the program source code, while continuations are influenced by function calls.

A sequence of statements that does not end in a terminator statement (a statement with no tail), such as a return statement or continue statement is treated as if it ended with a return statement that returned no values, i.e. return;.

The remainder of this section enumerates each kind of statement.

1.3.9.1 Return statements

return 17;
return 3 + 4;
return f(2, false);
return;

A return statement, written return EXPR;, where EXPR is an expression evaluates to the same value as EXPR. As a special case, return; is interpreted the same as return [];, that is, it evaluates to zero values.

A return statement is a terminator statement, so it must have an empty tail. For example,

{ return 1;
  return 2; }

is invalid, because the first return’s tail is not empty.

1.3.9.2 Identifier definition statements

If a value definition or function definition occurs in a statement position, then the identifiers are bound in the statement’s tail. For example,

const [ x, y ] = [ 3, 4 ];
const z = x + y;
return z;

evaluates to 7. And,

function f(x) { return x+1; };
return f(1);

evaluates to 2.

1.3.9.3 Conditional statements

if ( 1 + 2 < 3 ) {
  return "Yes!";
} else {
  return "No, waaah!"; }

A conditional statement, written if COND TRUE else FALSE, where COND is an expression which evaluates to a boolean and TRUE and FALSE as statements (potentially block statements), selects between the TRUE statement and FALSE statement based on whether COND evaluates to true.

Both TRUE and FALSE have empty tails, i.e. the tail of the conditional statement is not propagated. For example,

if ( x < y ) {
  const z = 3; }
else {
  const z = 4; }
return z;

is erroneous, because the identifier z is not bound outside the conditional statement.

A conditional statement may only include a consensus transfer in TRUE or FALSE if it is within a consensus step.

1.3.9.4 Block statements

A block statement is when a block occurs in a statement position, then it establishes a local, separate scope for the definitions of identifiers within that block. In other words, the block is evaluated for effect, but the tail of the statements within the block are isolated from the surrounding tail. For example,

const x = 4;
return x;

evaluates to 4, but

{ const x = 4; }
return x;

is erroneous, because the identifier x is not bound outside the block statement.

1.3.9.5 Expression statements

4;
f(2, true);

An expression, E, in a statement position is equivalent to the block statement { return E; }.

1.3.9.6 Local step

Alice.only(() => {
  const pretzel = random(); });

A local step statement is written PART.only(() => BLOCK), where PART is a participant identifier and BLOCK is a block. Any bindings defined within the block of a local step are available in the statement’s tail as new local state. For example,

Alice.only(() => {
  const x = 3; });
Alice.only(() => {
  const y = x + 1; });

is a valid program where Alice’s local state includes the private values x (bound to 3) and y (bound to 4). However, such bindings are not consensus state, so they are purely local state. For example,

Alice.only(() => {
  const x = 3; });
Bob.only(() => {
  const y = x + 1; });

is an invalid program, because Bob does not know x.

1.3.9.7 Consensus transfers

Alice.publish(wagerAmount)
     .pay(wagerAmount)
     .timeout(DELAY, () => {
       Bob.publish();
       commit();
       return false; }); 

Alice.publish(wagerAmount)
     .pay(wagerAmount)
     .timeout(DELAY, closeTo(Bob, false));

A consensus transfer is written PART.publish(ID_0, ..., ID_n).pay(PAY_EXPR).timeout(DELAY_EXPR, TIMEOUT_EXPR), where PART is a participant identifier, ID_0 through ID_n are identifiers for PART’s public local state, PAY_EXPR is a public expression evaluating to an amount of network tokens, DELAY_EXPR is a public expression that depends on only consensus state and evaluates to a time delta represented by a natural number, TIMEOUT_EXPR is an expression that evaluates to a timeout represented by a zero-arity function, which will be executed after DELAY_EXPR units of time have passed from the end of the last consensus step without PART executing this consensus transfer. The tail of a consensus transfer statement is a consensus step, which is finalized with a commit statement, which must occur in the tail.

The publish component exclusive-or the pay component may be omitted, if either there is no publication or no transfer of network tokens to accompany this consensus transfer. The timeout component may always be omitted. For example, the following are all valid:

Alice.publish(coinFlip);

Alice.pay(penaltyAmount);

Alice.publish(coinFlip)
     .timeout(DELAY, closeTo(Bob, false));

Alice.pay(penaltyAmount)
     .timeout(DELAY, () => {
       Bob.publish();
       commit();
       return false; });

1.3.9.8 Commit statements

commit();

A commit statement, written commit();, commits to statement’s continuation as the next step of the DApp computation. In other words, it ends the current consensus step and allows more local steps.

1.3.9.9 While statements

var [ heap1, heap2 ] = [ 21, 21 ];
invariant(balance() == 2 * wagerAmount);
while ( heap1 + heap2 > 0 ) {
  ....
  [ heap1, heap2 ] = [ heap1 - 1, heap2 ];
  continue; }

A while statement may occur within a consensus step and is written:

var [ VAR_0, ..., VAR_n ] = INIT_EXPR;
invariant(INVARIANT_EXPR);
while( COND_EXPR ) BLOCK

where the identifiers VAR_0 through VAR_n are bound to the result of the expression INIT_EXPR, which must evaluate to n values, and INVARIANT_EXPR is an expression, called the loop invariant, that must be true before and after every execution of the block BLOCK, and if COND_EXPR is true, then the block executes, and if not, then the loop terminates and control transfers to the continuation of the while statement. The identifiers VAR_0 through VAR_n are bound within INVARIANT_EXPR, COND_EXPR, BLOCK, and the tail of the while statement.

1.3.9.10 Continue statements

[ heap1, heap2 ] = [ heap1 - 1, heap2 ];
continue;

A continue statement may occur within a while statement’s block and is written:

[ VAR_0, ..., VAR_n ] = UPDATE_EXPR;
continue;

where the identifiers VAR_0 through VAR_n are the variables bound by the nearest enclosing while statement and UPDATE_EXPR is an expression which evaluates to n values.

A continue statement is a terminator statement, so it must have an empty tail.

1.3.10 Expressions

There are a large variety of different expressions in Reach programs.

The remainder of this section enumerates each kind of expression.

—
X
Y
Z

An identifier, written ID, is an expression that evaluates to the value of the identifier is bound to.

—
10
0xdeadbeef
007
true
false
"reality bytes"
'it just does'

A literal value, written VALUE, is an expression that evaluates to the given value. Numbers may be written in decimal, hexadecimal, or octal. Booleans may be written as true or false. Byte strings may be written between double or single quotes (with no distinction between the different styles) and use the same escaping rules as JavaScript.

—
! a
- a

A unary expression, written UNAOP EXPR_rhs, where EXPR_rhs is an expression and UNAOP is one of the unary operators: ! -.

Since all numbers are non-negative in Reach, the - iunary operator is useless.

It is invalid to use unary operations on the wrong types of values.

—
a && b
a || b
a + b
a - b
a * b
a / b
a % b
a | b
a & b
a ^ b
a << b
a >> b
a == b
a != b
a === b
a !== b
a > b
a >= b
a <= b
a < b

A binary expression, written EXPR_lhs BINOP EXPR_rhs, where EXPR_lhs and EXPR_rhs are expressions and BINOP is one of the binary operators: && || + - * / % | & ^ << >> == != === !== > >= <= <. The operators == and != operate on numbers, while the operators === and !== operate on byte strings. Numeric operations, like + and >, only operate on numbers. Since all numbers in Reach are integers, operations like / truncate their result. Boolean operations, like &&, only operate on booleans. It is invalid to use binary operations on the wrong types of values.

—
(a + b) - c

An expression may be parenthesized, as in (EXPR).

—
[ ]
[ 1, 2 + 3, 4 * 5 ]

A multiple value expression, written [ EXPR_0, ..., EXPR_n ], is an expression which evaluates to n values, where EXPR_0 through EXPR_n are expressions that evaluate to one value.

—
map[3]

An array reference, written ARRAY_EXPR[IDX_EXPR], where ARRAY_EXPR is an expression that evaluates to statically sized array and IDX_EXPR is an expression that evaluates to a natural number which is less than the size of the array, selects the given element of the array.

—
choosesFirst ? [ heap1 - amount, heap2 ] : [ heap1, heap2 - amount ]

A conditional expression, written COND_E ? TRUE_E : FALSE_E, where COND_E, TRUE_E, and FALSE_E are expressions, selects between the values which TRUE_E and FALSE_E evaluate to based on whether COND_E evaluates to true.

—
(() => 4)
((x) => x + 1)
((x) => { const y = x + 1;
          return y + 1; })

A lambda expression, written (ID_0, ..., ID_n) => EXPR, where ID_0 through ID_n are identifiers and EXPR is an expression, evaluates to an function which is an abstraction of EXPR over n values.

—
transfer(10).to(Alice)

A transfer expression, written transfer(AMOUNT_EXPR).to(PART), where AMOUNT_EXPR is an expression that evaluates to a natural number and PART is a participant identifier, performs a transfer of network tokens from the contract to the named participant. AMOUNT_EXPR must evaluate to less than or equal to the balance of network tokens in the contract account. A transfer expression may only occur within a consensus step.

—
interact.notify(handA, handB);
is(uint256, interact.chooseAmount(heap1, heap2))

An interaction expression, written is(TYPE, interact.METHOD(EXPR_0, ..., EXPR_n)), where TYPE is a type, METHOD is an identifier, and EXPR_0 through EXPR_n are expressions that evaluate to one value, evaluates to the result of an interaction with a frontend that receives the evaluation of the n expressions and sends a value of type TYPE if it is honest. The is component may be omitted, in which case it is treated as though it were is(boolean, ....).

—
assert( amount <= heap1 )
step( moveA )
digest( coinFlip )
random()
declassify( _coinFlip )

A function application, written EXPR_rator(EXPR_rand_0, ..., EXPR_rand_n), where EXPR_rator and EXPR_rand_0 through EXPR_rand_n are expressions that evaluate to one value. EXPR_rator must evaluate to an abstraction over n values or a primitive of arity n.

1.3.11 Standard Library

Reach’s standard library includes a few built-in primitives documented in this section.

—
assert( claim )

A static assertion which is only valid if claim always evaluates to true.

The Reach compiler will produce a counter-example (i.e. an assignment of the identifiers in the program to falsify the claim) when an invalid claim is provided. It is possible to write a claim that actually always evaluates to true, but for which our current approach cannot prove always evaluates to true; if this is the case, Reach will fail to compile the program, reporting that its analysis is incomplete. Reach will never produce an erroneous counter-example.

—
assume( claim )

An assumption where claim evaluates to true with honest frontends.

—
require( claim )

An requirement where claim evaluates to true with honest participants.

—
possible( claim )

A possibility assertion which is only valid if it is possible for claim to evaluate to true with honest frontends and participants.

—
digest( arg_0, ..., arg_n )

The digest primitive performs a cryptographic hash of the binary encoding of the given arguments, using the Keccak256 algorithm.

—
balance()

The balance primitive returns the balance of the contract account for the DApp.

—
random()

The random primitive returns a random unsigned integer of 256 bits. This primitive may not be called in consensus steps.

—
declassify( arg )

The declassify primitive performs a declassification of the given argument.

—
implies( x, y )

Returns true if x is false or y is true.

—
ensure( pred, x )

Makes a static assertion that pred(x) is true and returns x.

—
makeCommitment( x )

Returns two values, [ commitment, salt ], where salt is a random uint256, and commitment is the digest of salt and x.

—
checkCommitment( commitment, salt, x )

Makes a requirement that commitment is the digest of salt and x.

—
closeTo( Who, value )

Returns a function which accepts no arguments, has participant Who make a publication, then transfer the balance() to Who and end the DApp with the result value.
